# 项目提报材料：正职工考勤日晾晒报表自动化

## 一、项目概述（STAR 法则）

### S（Situation）— 场景与痛点

业务场景：美区正式工日考勤晾晒是每日必出的管理看板，需要从系统导出"每日打卡工时推送模版"和"美区签字报表"两份 Excel，按固定口径完成清洗、指标计算、汇总透视、明细整理，最终发布给各大区负责人。

原流程痛点：

1. **耗时长**：每次从导出到发布需 **2–3 小时**（数据导出整理 30–45 分钟、清洗与公式核算 45–75 分钟、透视汇总与文案撰写 30–45 分钟、校对发布 15 分钟）；
2. **口径不一致**：排班正确的判断规则（±1 小时、节假日特例、居家办公覆盖、首末卡完整性）分散在多份历史文档和经验中，不同人操作容易出现口径偏差导致返工；
3. **静态交付**：最终产出是静态 Excel 截图或表格，管理者无法按区域/部门/人员快速下钻，遇到异常需反复沟通确认；
4. **交付稳定性差**：依赖个人完成，一旦经办人请假或交接，产出时效和质量波动明显。

### T（Task）— 目标

建立一套**可复用、规则固化、交互可视**的自动化报表流程，使每日考勤晾晒从"手工制作"变为"一键生成"：

1. 完整覆盖核心指标：排班率、排班正确率、HUB 排班正确率、缺卡率、日超 8h 人数；
2. 汇总表按"大区→四级部门 / 中后台→三级部门"归类，含红黄绿灯标识和前一日趋势对比；
3. 明细表支持搜索、分页、收起展开；
4. 支持公网在线上传生成（无需本地环境）；
5. 支持 GitHub Pages 静态发布，团队可直接浏览器访问。

### A（Action）— 方案与落地

#### 方案架构

- **数据源精简**：从原来 4 份 Excel 精简为 2 份（主明细 + 签字报表），签字报表仅用于累计总工时合并，降低输入复杂度；
- **规则引擎化**：将排班正确判断的 6 条分支逻辑（空班次/±1h/节假日/首末卡完整/流程备注/居家办公覆盖）编码为确定性函数，消除人工理解偏差；
- **指标标准化**：
  - 排班率 = 有排班人数 / 总人数
  - 排班正确率 = 排班正确人数 / 总人数
  - 缺卡率 = 缺卡次数之和 / (应打卡人数 × 4)，休息日与提交流程人员不计入分母
  - 日超 8h 人数 = 每日总工时 + 居家办公（审批中） - 8 > 0 的人数
  - 本周加班工时 = max(累计总工时 - 40, 0)
- **交互看板**：生成响应式 HTML 报表，包含 KPI 卡片（涨跌箭头）、区域汇总表（红黄绿灯 + 前一日对比列）、明细分页表（搜索/收起）；
- **多渠道交付**：
  - 本地命令行一键生成（`python3 generate_report.py`）
  - 公网 Streamlit 网站上传生成（`app.py`）
  - GitHub Pages 静态发布（`docs/`）

#### 关键落地步骤

1. 梳理并结构化规则文档，将"自然语言口径"转化为"可执行判断函数"；
2. 按 `工号 + 考勤日期` 建立主明细与签字报表的关联键，合并累计总工时；
3. 实现排班正确判断的完整分支（含居家办公覆盖校验）；
4. 按"大区→四级部门 / 中后台→三级部门"实现分组汇总，增加前一日趋势对比与红黄绿灯；
5. 输出 HTML + JSON + CSV 三种格式，覆盖"看、分析、下载"三类需求；
6. 封装为 Streamlit 网页应用，支持公网无环境依赖访问；
7. 准备 `docs/` 目录和一键部署脚本，支持 GitHub Pages 静态发布。

### R（Result）— 结果与价值（量化可衡量）

#### 质量提升

- 排班正确判断从"人工经验"变为"6 条确定性规则"，口径偏差归零；
- 缺卡分母精确剔除休息日与流程人员，避免虚高缺卡率误导管理判断；

## 二、按提报要求的三问三答

### 1）我原本做这项工作多长时间 / 有什么痛点

每天制作一次正式工考勤日晾晒报表，原流程耗时 **2–3 小时**：

环节｜耗时
从系统导出 Excel 并整理格式｜30–45 分钟
清洗数据 + 套公式计算指标｜45–75 分钟
透视汇总 + 撰写区域点评文案｜30–45 分钟
校对 + 截图/格式化 + 发布｜15 分钟

核心痛点：
1. 每天重复同样的跨表操作（复制/筛选/粘贴/套公式），**机械劳动占比超过 80%**；
2. 排班正确的判断规则有 6 种情形（空班次、±1h、节假日无卡、首末卡不完整、流程备注、居家办公覆盖），**纯靠人工记忆容易遗漏**，返工率约 15%–20%；
3. 最终输出是静态表格，管理者问"某个站点谁缺卡了"还需要**再次回到 Excel 手动筛选**，沟通成本高。

### 2）我通过它给我节省了多长时间 / 解决了什么痛点

现在每次只需：导出 2 份 Excel → 上传或运行脚本 → **5–10 分钟**得到完整报表。

**直接节省：**
- 单次节省 **110–170 分钟**
- 按月 22 个工作日估算，节省 **42–62 小时/月**
- 效率提升 **93%–95%**

**解决的痛点：**
1. 手工透视与重复计算完全消除，从"每天做 2 小时"变为"每天点一下"；
2. 排班正确的 6 条规则写入代码，**口径一致性从"靠人记"变为"靠系统保证"**，返工归零；
3. 报表内置搜索/筛选/分页和红黄绿灯，管理者自助定位异常，**沟通确认环节压缩 90%**；
4. 支持公网访问和 GitHub Pages 发布，**不依赖经办人本机环境**，交接成本趋近于零。

### 3）我是怎么落地的

步骤｜动作｜产出
第一步｜梳理规则文档，将自然语言口径转化为 6 条排班正确判断函数和 4 条指标公式｜结构化规则清单
第二步｜精简数据源从 4 份到 2 份，按 `工号+考勤日期` 建立主键合并签字报表累计总工时｜标准化数据流
第三步｜编码实现清洗、指标计算、分组汇总（大区→四级/中后台→三级）、前一日趋势对比｜`generate_report.py`
第四步｜输出响应式 HTML 看板（KPI 卡片 + 红黄绿灯汇总 + 明细分页 + 搜索筛选）｜`attendance_report.html`
第五步｜封装为 Streamlit 网页应用，支持在线上传 → 生成 → 预览 → 下载｜`app.py`

## 三、可持续价值

- **可复制**：每天替换 2 份 Excel 即可复用，无需修改代码；
- **可扩展**：可继续接入"周报/月报汇总"、"班次异常预警推送"、"区域目标达成看板"；
- **可交接**：全部逻辑在代码中，配合 README 口径说明，新人 10 分钟即可上手操作；
- **可共享**：支持 Streamlit 公网部署和 GitHub Pages 静态发布，团队无需安装任何软件即可访问。
